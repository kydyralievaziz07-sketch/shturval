# Для 1С-программиста: вопросы по выгрузке (Yaros DataGate / HTTP-сервис)

Здравствуйте! Есть несколько моментов по HTTP-сервису выгрузки (`yarosDataGate`),
которые ломают работу нашего приложения. Опишу простыми словами.

---

## 1. Метод `/goods` падает по памяти — нужна выгрузка не «всё разом»

**Что происходит.** Метод `/goods` отдаёт **весь каталог разом** (~23 000 товаров,
~20 МБ одним ответом). 1С на этом периодически выдаёт ошибку:

> HTTP 500: «Ошибка получения товаров по причине: … Ошибка выполнения запроса:
> **Недостаточно памяти для получения результата запроса к базе данных**»

После этого сервис нередко «зависает» и перестаёт отвечать вообще (таймауты на все
методы), пока сам не восстановится. Из-за этого каталог товаров у нас периодически
недоступен.

Нам нужно одно из двух (А — предпочтительно, Б — как минимум):

### Вариант А (ЛУЧШИЙ): отдавать только то, что ИЗМЕНИЛОСЬ с прошлого раза

Мы храним каталог у себя в базе и хотим **обновлять только изменения**, а не качать
20 МБ каждый раз. Для этого нужен параметр «изменено после»:

- `/goods?changedSince=2026-06-24T15:30:00` — вернуть **только товары, у которых
  поменялись цена / остаток / любые поля ПОСЛЕ указанного момента** (плюс новые товары).

Тогда мы раз в пару минут запрашиваем крошечную порцию изменений, обновляем свою базу
по `ID` товара — и 1С больше **никогда не собирает весь каталог** и не падает по памяти.
Это самый лёгкий для сервера режим. (Полную выгрузку делаем редко — только на первый
раз и иногда для сверки.)

### Вариант Б (как минимум): отдавать каталог ЧАСТЯМИ (пагинация)

Если «изменено после» сделать сложно — хотя бы пагинацию, чтобы не отдавать всё разом:

- `/goods?limit=1000&offset=0`, `offset=1000`, `offset=2000` … (порциями по 1000–2000),
  **или**
- `/goods?categoryId=<ID>` — товары по одной категории за запрос (id из `/categories`).

**Желательно** в ответ добавить общее число товаров (`total`).

## 2. Нужна ФАКТИЧЕСКАЯ себестоимость остатка по каждому товару

**Зачем.** Мы показываем «стоимость склада по себестоимости». Сейчас в `/goods` в
массиве `PRICES` есть тип **«Закупочная»** — но это **цена из прайса** (текущая), а не
фактическая себестоимость, по которой реально пришли остатки. Из-за этого наша сумма
склада расходится с тем, что 1С показывает в отчёте «по себестоимости» (у нас выходит
завышенно, т.к. закупочные цены со временем менялись).

**Что просим.** Добавить в выгрузку товара поле с **фактической себестоимостью остатка**
— той самой, по которой 1С считает стоимость остатков в отчётах (средневзвешенная / по
партиям). Например, поле `COST_ACTUAL` (себестоимость за единицу) или
`STOCK_COST_TOTAL` (себестоимость всего остатка по товару).

Тогда стоимость склада в нашем приложении будет совпадать с 1С **до сома**.

---

## 3. Проверьте память/стабильность самого сервера 1С

В моменты сбоя 1С пишет «**недостаточно памяти**» и зависает целиком (не отвечает на
все методы, пока сама не поднимется). Со своей стороны мы уже **снизили частоту своих
запросов в ~15 раз** (тяжёлый каталог теперь дёргаем редко), чтобы не нагружать сервер.

Просьба проверить на стороне 1С:
- **Чем именно занята память в момент сбоя** — наши запросы, чужие интеграции (сайт/бот),
  фоновые операции, тяжёлые отчёты? Кто реально съедает память?
- **Хватает ли серверу 1С оперативной памяти (RAM)** под текущий объём данных, и не
  растёт ли потребление со временем (возможная утечка / незакрытые сессии)?

Это поможет понять, достаточно ли только доработки выгрузки (пункт 1) или серверу нужно
ещё добавить памяти.
